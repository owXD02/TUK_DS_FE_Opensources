## 수정이 "필요할 뻔한" 지점 (실제 수정은 하지 않음 — 사용자 지시)

아래는 13개 테스트는 통과했지만, **문제 문구가 조금만 달라지면** 라이브러리 수정 또는
비단순 커스텀 코드가 필요해지는 잠재 위험 지점이다. 시험 전에 인지만 해두면 된다.

### [위험 1] 구조체(레코드) 정렬을 "선택/삽입/버블"로 명시 요구할 때  ★가장 가능성 높음
- 증상: `sort.h`의 정렬 함수는 시그니처가 `int a[]` 전용이다. 구조체 배열은 못 받는다.
- Q8에서는 **heap의 tag 기능으로 우회**(학점=키, 배열인덱스=tag → 꺼내면 학점순)했다.
  하지만 문제가 "**선택정렬로** 학생 구조체를 학점순 정렬하라"라고 알고리즘을 못 박으면 우회 불가.
- 왜 수정이 필요한가: 그 경우 `drone.c`의 `sortDronesBy`처럼 **구조체 전용 정렬 함수를 새로
  작성**해야 한다(README 레시피 D에 "필드 이름만 바꿔 복사" 안내는 있으나, 이는 순수 조립이
  아니라 코드 작성에 해당).
- 대비책(수정 없이): 시험장에서 `sortDronesBy` 본문을 복사해 구조체/필드명만 바꿔 쓴다.
  README 레시피 D가 그 출발점이다.

### [위험 2] 학번 등 숫자 키가 int 범위를 넘는 경우
- 증상: `bst.h`의 key, `heap.h`의 data가 `int`다. 테스트의 학번(20xxxxxxxx, 약 20억)은
  int 최대(약 21.4억) 안이라 통과했다. 그러나 11자리 학번이나 큰 수가 오면 오버플로.
- 왜 수정이 필요한가: 그땐 bst/heap의 `int`를 `long`/`long long`으로 바꿔야 한다(=라이브러리 수정).
- 대비책(수정 없이): 학번을 키로 쓰지 말고 "배열 인덱스(0..n-1)"를 키/노드로 쓰고,
  실제 학번은 별도 배열에 저장해 인덱스로 참조. 대부분의 문제는 이렇게 우회 가능.

### [위험 3] BST/트리 노드에 레코드 전체(이름·나이 등)를 담아 되돌려 받아야 할 때
- 증상: `bst.h`의 노드는 `int key`만 있고 부가정보(satellite data) 필드가 없다.
- Q4(시험예상문제15 유형)는 "학번으로 검색 → 학번.txt 열기"라서, 검색 결과의 key(학번)만으로
  파일명을 만들어 통과했다. 즉 **부가정보가 필요 없어서** 문제가 안 됐다.
- 왜 수정이 필요한가: "이름으로 검색 → 그 학생의 학점을 트리에서 바로 출력"처럼 노드가
  레코드를 들고 있어야 하면 노드 구조체에 필드 추가가 필요(=라이브러리 수정).
- 대비책(수정 없이): heap처럼 BST도 "키 + 배열인덱스" 2개를 같이 다루면 되지만 현재 bst엔
  tag가 없다. 따라서 **검색은 key로 하고, 부가정보는 별도 레코드 배열에서 key로 다시 찾기**
  (병렬 배열 방식)로 우회. Q4에서 실제로 이 방식이 통했다.

### [위험 4] "검색 시간"을 비교 횟수로 요구할 때
- 증상: Q4에서 `clock()`으로 측정했더니 0 tick(너무 빨라 측정 불가)이었다.
- `bstSearch`는 비교 횟수를 반환하지 않는다. 출제자가 "탐색 비교 횟수"를 원하면 라이브러리만으론 안 됨.
- 대비책(수정 없이): main에서 직접 `while`로 트리를 내려가며 카운트하는 짧은 탐색 루프를
  손으로 작성(ch10 binarySearch의 cnt 패턴과 동일). 라이브러리 수정 불필요.

## 결론
- 예측 가능한 큐~정렬 문제는 **현재 코드만으로 거의 전부 조립 해결**된다(13/13).

  이는 `sortDronesBy` 복사로 대응한다(README 레시피 D).
- 나머지는 모두 "병렬 배열/인덱스" 우회로 수정 없이 풀 수 있음을 실증함.
